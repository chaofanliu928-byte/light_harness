# Writing-plans 阶段治理规则

> 当 Superpowers 的 writing-plans skill 激活时，读取本文件。

## 反模式约束(用户 feedback 硬编码 — 必读)

> 依据:`memory/MEMORY.md` 索引下的 feedback 条目。

- **实战验证不阻塞 harness 开发**(`feedback_realworld_testing_in_other_projects.md`):涉及"实战留痕 / 真实场景验证 / meta-L4"等延后项,**不**为补这些数据在 harness 自仓库构造 artificial trial。明确 documented 推后到 P1 真实项目阶段产出。决策时不把"等实战数据"当 blocking 条件;采用最佳已知判断 + 推真实项目验证。

## 前置：读取设计文档

1. **读取设计文档** `docs/superpowers/specs/[功能名]-design.md`
2. 计划中的模块划分必须与设计文档第 2 节一致
3. 计划中的接口实现必须与设计文档第 3 节的接口定义一致
4. **涉及前后端的接口任务必须以"先创建/更新共享类型契约文件"作为第一步**（设计文档第 3.3 节）
5. 计划中的数据模型必须与设计文档第 4 节一致
6. **如果计划需要偏离设计文档，必须先回到系统设计阶段修改设计，不要在计划中静默偏离**

## 架构合规

- 计划中的所有文件路径必须符合 `docs/ARCHITECTURE.md` 的分层规则
- 任何跨层依赖必须在计划阶段标出并说明理由
- 新模块必须放在架构规定的目录下

## 模块文档

- 每个任务的实现步骤中必须包含"更新涉及模块的 README.md"
- 新建模块的任务必须包含"按 `docs/references/MODULE_DOC_TEMPLATE.md` 创建 README.md"
- 模块文档和代码在同一个 commit 里提交

## 计划粒度：两类任务，两种写法

计划中的任务分为两类，写法不同：

### 契约任务（指令式）

创建/更新接口、类型定义、共享契约等**跨模块/跨角色的声明**。这类任务必须精确，因为接口是双方的合同。

写法——**给出精确的定义**：

```markdown
任务 1：创建共享类型契约
- 操作：按设计文档 3.3 节，创建/更新 shared/types/[模块].ts
- 包含类型：[逐个列出类型名称和字段]
- 依据：设计文档第 3.3 节
```

契约任务永远排在实现任务之前（已有规则：涉及前后端的接口任务必须以"先创建/更新共享类型契约文件"作为第一步）。

### 实现任务（问题式）

模块内部的逻辑实现。这类任务描述**要解决的问题和约束**，不规定具体实现路径——实现 agent 比规划者更接近代码的实际状态，由它决定最佳实现方式。

写法——**问题 + 约束 + 验证标准**：

```markdown
任务 3：解决 [问题名称]
- 问题：[要解决什么——用业务语言描述]
- 约束：
  - 接口遵循 shared/types/[模块].ts 中的定义（设计文档 3.3 节）
  - 放在 src/[目录]/ 下（ARCHITECTURE.md 分层规则）
  - [其他来自设计文档或 RUBRIC 的约束]
- 验证标准：[怎么证明这个任务做完了——具体的测试场景或可观测的行为]
```

**约束的来源和粒度：**

| 约束类型 | 来自哪里 | 必须写到任务中 |
|---------|---------|--------------|
| 接口/类型契约 | 设计文档 3.3 节 | ✅ 引用具体的共享类型文件 |
| 模块边界/文件位置 | ARCHITECTURE.md + 设计文档第 2 节 | ✅ 指定目录，不指定文件名 |
| 行为规则 | implementation-rules.md | ❌ 不需要重复写——实现 agent 会自行读取治理文件 |
| 内部实现细节 | — | ❌ 不写——留给实现 agent 判断 |

### 通用要求

- 每个任务标注它涉及哪些已有模块（参考 `docs/active/module-registry.md`，如存在）
- 复用已有模块的接口，不要重复实现

## 测试计划

- 基于设计文档第 5 节（边界条件）编写测试场景
- 每个核心场景（设计文档第 1.2 节）必须有对应的测试任务
- 标注测试层级（单元/集成/E2E）和 mock 策略
