# Brainstorming 阶段治理规则

> 当 Superpowers 的 brainstorming skill 激活时，读取本文件。
> 本阶段的核心职责是**需求对接**，不是直接跳到方案设计。
> AI 必须对需求的完整性和清晰度负责。

## 阶段一：准备

1. **读取 `docs/RUBRIC.md`** — 记住惩罚项和奖励项
2. **读取 `docs/ARCHITECTURE.md`** — 记住分层规则
3. **检查 `docs/references/`** — 读取相关参考资料
4. **运行 `/session-search`** — 搜索历史上下文，静默跳过无命中
5. 如果 session-search 返回了与当前需求冲突的历史决策，先向用户呈现，确认是否影响当前方向

## 阶段二：需求深挖

> 不要在用户说完第一句话后就开始设计。先确保真正理解了需求。
> 判断标准：如果换一个工程师来读这个需求描述，他实现的东西是否一定和你理解的一样？如果不一定，就需要继续挖。

6. **识别模糊点** — 用户描述中哪些词有多种理解方式？
7. **识别缺失点** — 逐项检查非功能性需求：性能？安全？可用性？可维护性？数据量级？
8. **识别冲突点** — 用户的要求之间、或与 RUBRIC/ARCHITECTURE 之间是否矛盾？
9. **识别隐含假设** — AI 自己是否做了用户没说过的假设？
10. **确认优先级** — 如果有多个场景，哪些是 MVP 必须的？哪些可以后做？

对以上问题，**向用户提问**，不要自行假设答案。

**收敛判断**：当每个核心场景都能写出"谁 → 做什么 → 系统做什么 → 看到什么"的完整描述时，深挖可以结束。

## 阶段三：需求确认清单

> 清单的字段结构与 DESIGN_TEMPLATE.md 第 1 节对齐，便于直接流转。

11. 将用户的回答整合为一份**需求确认清单**：

```markdown
## 需求确认清单

### 1.1 用户目标
[用户想实现什么——用用户的语言，不是技术语言]

### 1.2 核心场景（按优先级排序）
1. [P0] [场景]：[谁] → [做什么] → [系统做什么] → [看到什么]
2. [P1] ...

### 1.3 边界与约束
- 做什么：[范围内]
- 不做什么：[明确排除]
- 性能要求：[如有]
- 安全要求：[如有]
- 兼容性要求：[如有]

### 1.4 关联需求
- 依赖的已有功能：[列出]
- 被哪些未来功能依赖：[如已知]

### 已确认的决策
- [问题]：[用户的回答]

### RUBRIC 风险标记
- 涉及的惩罚项：[列出哪些惩罚项与本需求相关]
- 涉及的奖励项：[列出哪些奖励项可以体现]
（具体应对方式留到系统设计阶段填写）
```

12. **自检（第一轮）**：
    - 清单覆盖了用户说过的所有要点？
    - "不做什么"是否至少列了一个用户可能误以为在范围内但实际不做的事？
    - 有没有 AI 偷偷加入用户没提过的需求？
    - RUBRIC 惩罚项是否都已标记？

13. **把清单给用户确认**。用户可能补充或修改。

14. **自检（第二轮）**：用户的补充是否引入了新的矛盾或模糊？如果是，回到阶段二（步骤 6-10）重新检查变更部分。
    - **收敛保障**：如果已经经历了 3 轮确认循环，向用户说明剩余的开放项，建议锁定当前版本，在设计阶段遇到具体问题时再回来补充。

## 阶段四：方案讨论

> 需求确认清单锁定后才进入方案讨论。

15. 提出 1-2 个可行方案，说明各自的优劣和对 RUBRIC 的影响
16. 方案必须回应 RUBRIC 中的惩罚项（主动避免）和奖励项（主动追求）
17. 如果用户想法会触发惩罚项，坦率指出
18. 确定方案方向后，简要记录选择理由

## 输出

- 创建设计文档 `docs/superpowers/specs/[功能名]-design.md`，填写第 1 节（需求摘要）
- 方案方向记录在设计文档或 `docs/superpowers/specs/` 中
- 同步更新 `docs/product-specs/index.md` 功能索引

## RUBRIC 渐进式积累

在整个 brainstorming 过程中，如果用户表达了明确的偏好或反感：

- "不要这样做" / "这个方向不对" → 整理为 RUBRIC.md 惩罚项，用 Edit 追加
- "就要这样" / "这个思路好" → 整理为 RUBRIC.md 奖励项，用 Edit 追加
- 影响全局的决策（如"本项目统一用 XXX"）→ 同时写入 RUBRIC.md 和 ARCHITECTURE.md

写入后简短告知用户："已记录到项目标准中。"

## 回退入口（供其他阶段使用）

| 回退原因 | 从哪一步开始 |
|---------|------------|
| 需求清单有遗漏 | 阶段三步骤 11 |
| 需求之间有矛盾 | 阶段二步骤 6-10 |
| 方向性需要调整 | 阶段二步骤 6 |
