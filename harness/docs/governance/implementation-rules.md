# Implementation 阶段治理规则

> 当 Superpowers 的 subagent-driven-development 或 executing-plans 激活时，
> 子代理读取本文件。

## 评分标准

- 读取 `docs/RUBRIC.md`，遵守项目特定标准
- 主动避免惩罚项中列出的模式
- 主动体现奖励项中鼓励的做法

## 文档先行原则（变更时同样适用）

> 文档第一公民不只是"新建时先写文档"，变更时也是**先改文档再改代码**。

任何代码变更涉及以下文档时，必须按顺序执行：

| 变更类型 | 先改什么文档 | 再改什么代码 |
|---------|------------|------------|
| 接口/字段变更 | 设计文档第 3.3 节（类型契约） | 共享类型文件 → 前后端代码 |
| 模块职责变更 | 设计文档第 2 节（模块划分） | 模块代码 + README |
| 架构变更 | ARCHITECTURE.md | 目录结构 + 代码 |
| 模块内部变更 | 模块 README.md | 模块代码 |

**如果发现文档和实际需要不一致：先改文档，不要在代码中偷改然后事后补文档。**

## 类型契约

- 涉及前后端交互的代码，**必须从共享类型契约文件 import 类型，不能自己定义**
- 新增/修改 API 字段时，**先改共享类型文件，再改前后端代码**
- 如果发现设计文档的类型定义和实际需要不一致 → 回退到系统设计修改第 3.3 节

## 架构决策

遇到不确定的技术选择时：
1. 写入 `docs/decisions/[主题].md`（参照 `docs/decisions/_TEMPLATE.md`）
2. 标注状态为"🟡 待决定"
3. **停下来请求用户决定**，不要自行拍板

## 模块文档

- 每个模块的代码修改必须同步更新该模块的 README.md
- 接口变了 → 更新"对外接口"段落
- 依赖变了 → 更新"上下游依赖"段落
- 追加"变更历史"一行
- **先改文档，再改代码，两者在同一个 commit 中提交**
- PostToolUse hook 会提醒你是否遗漏了文档更新

## 实现行为约束

> 以下规则针对 LLM 编码时最常见的失误模式。

### 最小变更原则

- **只改任务要求的代码**。不优化相邻代码、不重写注释、不调整格式
- **匹配现有风格**，即使你会用不同写法
- 发现不相关的死代码或问题 → 在 PR 中提一句，不在本次变更中修复
- 自检：**每一行 diff 都能追溯到当前任务要求吗？** 不能的就撤掉

### 孤儿代码规则

- 你的变更导致某个 import/变量/函数不再被使用 → **你必须清理**
- 变更前就已经存在的死代码 → **不要动**，除非用户明确要求

### 简洁性自检

- 实现完成后自问：**一个资深工程师会说这写复杂了吗？**
- 如果是 → 重写，不要解释为什么需要复杂
- 禁止添加：未被要求的抽象层、"未来可能用到"的配置项、不可能触发的错误处理

## 提交规范

- 每次只做一个功能点，做完 `git commit` 再做下一个
- 提交信息格式：`feat: [功能名称] - [简要描述]`

## RUBRIC 渐进式积累

实现过程中如果用户纠正了 AI 的做法：
- 用户说"不要这样写" → 整理为 RUBRIC.md 惩罚项
- 用户说"以后都这样做" → 整理为 RUBRIC.md 奖励项
- 用 Edit 追加到 RUBRIC.md 的项目特定标准部分
- 简短告知用户："已记录到项目标准中，后续会遵守。"

## 回退判断

实现过程中如果发现以下情况，**不要在代码层面打补丁**，而是回退到对应阶段：

| 发现什么 | 回退到 | 带上什么 |
|---------|-------|---------|
| 接口签名和设计文档不一致 | 系统设计（修改第 3 节接口定义） | 具体哪个接口、实际需要什么 |
| 模块职责不清，一个模块在做两件事 | 系统设计（修改第 2 节模块划分） | 哪个模块、应该怎么拆 |
| 需求有场景设计里没覆盖 | brainstorming（补充需求确认清单） | 缺失的场景描述 |
| 纯代码 bug（逻辑错误、类型错误） | 不回退，在当前阶段修复 | — |

**判断辅助**：这个问题能否只通过修改当前文件的代码来解决？如果能，它是代码 bug。如果需要修改设计文档中的接口定义或模块职责才能根本解决，它是设计缺陷。

**回退时代码的处理**：
- 代码保留在功能分支上，不废弃
- 设计修复并通过审查后，在 planning 阶段为受影响的接口增加修改任务
- 回到实现阶段修改代码，不需要从头实现
