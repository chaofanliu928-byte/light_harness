# 系统设计阶段治理规则

> 在 brainstorming 完成（需求锁定）之后、writing-plans 之前读取本文件。
> 本阶段的核心职责是**产出自洽的系统设计文档**。
> AI 对设计的完整性和一致性负责。发现不自洽时在原地修复，不往下传递。

## 反模式约束(用户 feedback 硬编码 — 必读)

> 依据:`memory/MEMORY.md` 索引下的 feedback 条目。设计阶段易踩 4 条反模式,**进入本阶段时必须主动遵守**:

- **判断必须基于事实和逻辑**(`feedback_judgment_basis.md`):设计决策不得用"市场判断 / 别人项目数据"支撑。每条决策需可指事实证据。
- **不便利答案掩盖规范缺口**(`feedback_spec_gap_masking.md`):设计中识别的缺口(如"M3 hook 不可见")要**显式标"已知缺口"+ 给技术原因 + 给补救方向**,不要包装成"已处理"。
- **过度工程化指控必须做反向追问**(`feedback_dimension_addition_judgment.md`):挑战者或自检判"过度工程化"时,**必须**反向追问:"如果不用这个方式,那么我们怎么解决之前的问题?"。找不到替代解法 → 不是过度工程,是必要复杂度。
- **bootstrap 自举不可证不算缺陷**(`feedback_unprovable_in_bootstrap.md`):自举系统在自己落地前不可证(先有鸡还是先有蛋)**不算缺陷**,声明 + 推后续阶段实战验证即可;但 spec 内**具体可证**漏洞必须保留。

## 前置条件

- 需求确认清单已被用户确认（brainstorming 阶段输出）
- 方案方向已确定
- 设计文档骨架已创建：`docs/superpowers/specs/[功能名]-design.md`（第 1 节由 brainstorming 填写）

## 规模判断

在开始设计前，根据需求复杂度选择设计深度：

| 级别 | 判断标准 | 设计深度 | 前置硬条件(任一不满足升至标准级) |
|------|---------|---------|----------------------------------|
| **轻量级** | 改动 1-2 个文件，不涉及新模块或接口变更 | 写**精简版设计文档**：只填第 1 节（需求摘要）+ 第 8 节（涉及文件和改动说明）+ 第 9 节自洽性检查中适用的项。不需要 design-review | (1) 改动行数 < 100 行(`git diff --stat` 总和);(2) 不涉及 `docs/RUBRIC.md` 红线段(目录约定 / 命名规范 / 架构边界);(3) 不涉及多模块共用接口或类型;(4) spec §0 偏离说明不引 RUBRIC 维度做免审依据(与 finishing-rules.md `## 反模式约束` 段 M2 同步约束) |
| **标准级** | 涉及新模块或接口变更，但不超过 3 个模块 | 写完整设计文档，不适用的节可写"不适用" | — |
| **重量级** | 涉及 4 个以上模块或跨系统交互 | 写完整设计文档，所有节必填 | — |

**文档是第一公民。新建时先有文档再写代码，变更时先改文档再改代码。** 区别只在文档的厚度，不在有没有。

如果不确定级别，按**标准级**执行。

> **关于第 4 列前置硬条件的默认升级原则**(2026-04-29 meta-review D2-F4):4 条前置硬条件中,(1) `git diff --stat` 行数 + (4) grep `RUBRIC` 引用是机械可判;但 (2) "不涉 RUBRIC 红线段" + (3) "不涉多模块共用接口或类型" 实质是语义判断。**对 (2)(3) 任何疑义时,默认升至标准级**(与上方"如果不确定级别,按标准级执行"原则一致),不要靠 spec 自宣告"我觉得不涉及"绕过。

## spec §0 偏离规则

- spec §0 "偏离说明"只允许记录**结构差异**(用什么 heading / 用什么编号 / 调整哪节顺序),**不允许**用来豁免 design-review(2026-04-17 retrospective P2 #"spec §0 偏离说明 不能用来免 design-review")
- 任何非 emergency 的 spec **必须至少过一次 design-review**;偏离模板者反而应**加强 review,不是减轻**
- 唯一不需要 design-review 的路径:轻量级判定(本文件 §规模判断 表)— 由表中四列前置硬条件机械判定,**不**由 spec 自宣告
- emergency 路径定义:线上故障紧急修复 + 同时缺时间走 design-review;事后必须补 design-review,不能"emergency 一次永久免审"
  - **harness 自身仓库语境**(2026-04-29 meta-review D3-F4):harness feature scope 改动同样适用本路径;meta scope 改动走 meta-review 流程,不经本 emergency 路径(harness 自仓库无"线上故障"场景,本路径主要为下游业务项目设计)
- **同步约束**(2026-04-29 meta-review D2-F3):另见 `finishing-rules.md` `## 反模式约束` 段 RUBRIC 维度不作跳过依据条款(M2 同步约束)。M4 spec §0 偏离规则是 design 阶段**正向阻断**,M2 是 finishing 阶段**反向回顾**,两条规则协同覆盖完整时序

## 角色分离

本阶段由**独立的 designer agent** 执行（context: fork），不是调度者亲自做设计。

- **调度者**：检查前置条件 → 判断规模 → fork designer agent → 接收结果 → 处理待决策项 → fork design-reviewer agent → 处理审查结果
- **designer agent**：逐节填写设计文档 + 自检（详见 `.claude/agents/designer.md`）
- **design-reviewer agent team**：领审员 fork 4 个并行子智能体（自洽性/完整性/合理性/RUBRIC）独立审查，汇总判定（详见 `.claude/agents/design-reviewer.md`）

## 调度者的执行步骤

1. 确认前置条件满足（需求锁定 + 方案确定）
2. 判断规模级别（轻量/标准/重量）
3. fork designer agent，传入：设计文档路径、RUBRIC.md、ARCHITECTURE.md、规模级别
4. designer 返回后，检查是否有待用户决定的决策（docs/decisions/ 中的 🟡 项）
   - 有 → 向用户呈现选项，等用户决定后更新设计文档
   - 无 → 继续
5. 检查是否有需求缺陷发现
   - 有 → 回退到 brainstorming 对应步骤（见 brainstorming-rules.md 回退入口表）
   - 无 → 继续
6. 标准/重量级：运行 `/design-review`（fork design-reviewer agent）
   - 通过 → 设计文档锁定，进入 writing-plans
   - 不通过 → 将审查意见传给 designer agent 修复，修复后重新审查
   - **连续 2 次不通过 → 停下来和用户讨论设计方向**
7. 轻量级：不需要 design-review，设计文档直接锁定

## designer agent 的工作方式（参考，详见 agent 定义）

按 `docs/references/DESIGN_TEMPLATE.md` 逐节填写。保存到 `docs/superpowers/specs/[功能名]-design.md`。

> 逐节推进，每节自检。不通过就在原地修改，不进入下一节。

### 第 1 节：需求摘要

1. 从 brainstorming 的需求确认清单中提炼
2. **自检**：
   - 每个核心场景都有输入/处理/输出？
   - "不做什么"明确？
   - 和需求确认清单对得上，没有遗漏或偷换？

### 第 2 节：模块划分

3. 列出所有新建和改动的模块
4. 画出依赖关系
5. **自检**：
   - 每个模块只有一个明确职责？
   - 依赖方向符合 ARCHITECTURE.md？
   - 没有循环依赖？
   - 每个需求场景都能在模块中找到实现路径？

### 第 3 节：接口定义

6. 定义所有模块间接口和外部接口
7. **自检**：
   - 每个接口的双方都定义了？
   - 参数和返回类型存在于项目类型系统中？
   - 每个接口都有错误处理约定？
   - **交叉验证**：接口的入参/出参与第 1 节需求中的数据对得上？

### 第 4 节：数据模型

8. 定义数据实体、数据流、状态变更
9. **自检**：
   - 数据流中每一步的类型与接口定义一致？
   - 实体字段覆盖了所有接口使用的数据？
   - 状态机没有死状态或不可达状态？
   - **交叉验证**：数据模型和第 3 节接口完全对齐？

### 第 5 节：边界条件与错误处理

10. 列出边界场景和错误传播路径
11. **自检**：
    - 每个接口的错误情况都有对应处理？
    - 错误传播路径完整，不存在吞掉错误的地方？
    - **交叉验证**：边界条件覆盖了第 4 节数据模型中每个字段的极端值？

### 第 6 节：测试策略

12. 基于核心场景和边界条件定义测试方式
13. **自检**：
    - 每个核心场景都有对应测试？
    - 每个边界条件都有对应测试？
    - 测试层级选择合理？

### 第 7 节：设计决策

14. 记录已确定的决策，填写 RUBRIC 应对方式
15. 不确定的决策写入 `docs/decisions/`，标 🟡 待用户决定
16. **如果有阻塞性待决策（影响接口/数据/架构）→ 停下来问用户**

### 第 8 节：与既有系统的影响

17. 列出需要改动和需要验证兼容的已有文件
18. **自检**：
    - 改动已有文件时，所有调用方都考虑到了？
    - 新旧模块交互没有引入不兼容？

## 全局自洽性检查

> 所有节完成后，做一次全局检查。

19. 逐条检查设计模板第 9 节的自洽性清单：
    - 需求 ↔ 模块：每个需求场景都有模块实现路径？
    - 模块 ↔ 接口：每个模块的职责都通过接口体现？
    - 接口 ↔ 数据：接口中的数据类型都在数据模型中定义了？
    - 数据 ↔ 边界：每个数据字段的边界值都有处理？
    - 依赖 ↔ 架构：模块依赖方向符合 ARCHITECTURE.md？
    - 决策 ↔ 需求：设计决策没有偏离需求约束？
    - 决策 ↔ 架构：设计决策中的架构选择与 ARCHITECTURE.md 一致？
    - 影响 ↔ 模块：第 8 节改动的文件与第 2 节标记为"改动"的模块对应？
    - RUBRIC ↔ 设计：第 7 节对每个 RUBRIC 惩罚项都有应对方式？
    - 契约 ↔ 接口：第 3.3 节的共享类型覆盖了所有 API 端点？字段命名一致？

20. **不通过的项 → 回到对应的节修复，修复后重新做全局检查。**

## 全局检查通过后

21. 标准/重量级：运行 `/design-review`（fork design-reviewer agent team）
22. 审查通过 → 设计文档锁定，进入 writing-plans
23. 审查不通过 → 将审查意见传给 designer agent 修复，修复后重新审查
24. **连续 2 次不通过 → 停下来和用户讨论设计方向**
25. 轻量级：不需要 design-review，设计文档直接锁定

## Fork 失败降级

如果 agent fork 失败（超时、上下文溢出、工具不可用）：
- **designer fork 失败**：调度者按 DESIGN_TEMPLATE 自行完成设计（降级为非隔离模式）
- **design-reviewer fork 失败**：调度者按 design-reviewer.md 中的 4 个维度自行逐步审查

**补偿机制**：降级执行的结果在设计文档顶部标注 `> ⚠️ 降级执行 [日期] — 未经独立 agent 验证，下次会话须由独立 agent 重新审查`。下次会话 session-init 检测到此标注时提醒调度者重新 fork 对应 agent 验证。

## 回退条件

在系统设计过程中，如果发现以下情况，**回退到 brainstorming 重新对接需求**：

- 需求确认清单有遗漏（用户没确认的场景）
- 需求之间存在 brainstorming 阶段未发现的矛盾
- 设计无法满足 RUBRIC 的某个标准，需要和用户协商调整需求

回退时带上具体发现（"第 X 节自检时发现 Y"），不要空手回退。
回退入口：参考 brainstorming-rules.md 末尾的"回退入口"表，按原因选择从哪一步开始。

## 回退时设计文档的处理

- 设计文档**保留**，不废弃
- 在第 1 节顶部标注 `> ⚠️ 需求变更后待修订`
- brainstorming 完成后，从第 1 节开始重新走自检流程
- 未受变更影响的节可以保留（如果自检仍然通过）
