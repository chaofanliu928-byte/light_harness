# 多智能体审查指南

> 所有审查类 agent（evaluator、design-reviewer 等）设计时参照本文件。
> 本文件定义审查原则和操作模式，不定义具体维度——维度由每个 agent 根据审查对象自行设计。

## 核心架构：对抗-决策分离

```
子 agent（对抗者）：找具体问题，附证据
    ↓ 输出：问题清单 + 代码/文档证据
领审员（决策者）：基于问题清单做评分和决策
    ↓ 输出：分数 + 决策（通过/精磨/推翻等）
```

**找问题的和做判断的分开。** 子 agent 不打分，领审员不找问题。

为什么：子 agent 同时找问题和打分时，两件事都做不好——找问题时不够尖锐（因为要"客观"），打分时不够有据（因为没深入找）。分离后，对抗者可以尽情挑刺，决策者拿到充分证据后做有据可依的判断。

## 四个原则

### 1. 每个 agent 有独立视角，不重叠

每个子 agent 的审查范围必须排他——看不同的东西、问不同的问题。

| ❌ 错误分工 | ✅ 正确分工 |
|------------|-----------|
| Agent A：审查后端 | Agent A：审查设计逻辑（算法对不对） |
| Agent B：审查 Python 代码 | Agent B：审查代码集成（会不会破坏现有流程） |
| Agent C：全面审查 | Agent C：审查失败场景（边界、异常、恢复） |

维度怎么切取决于审查对象的性质，不存在通用模板。判断标准：**如果两个 agent 的报告有大量重叠，说明分工有问题。**

### 2. 让 agent "对抗"，不是"验证"

最大的陷阱：给 agent 一份方案说"看看这个方案对不对"，它会倾向于说好。

正确做法——给 agent 立场，让它找问题：

- ❌ "审查这份设计文档"
- ✅ "你是一个严苛的 [角色]。找出 3 个最可能让这个方案失败的地方，用代码/文档证据支持。"

关键词：**具体角色 + 具体数量 + 要求证据**。

### 3. 要求可验证的输出

不要让 agent 给出模糊的建议，要求它指向具体位置和具体场景。

- ❌ "建议改进错误处理"
- ✅ "在 keyframes.py:45 的 derive_keyframes() 中，当 shots 为空列表时会导致 IndexError"

子 agent 的输出格式应固定为：

```markdown
### [问题标题]
- **位置**：[文件:行号 或 文档:章节]
- **问题**：[具体描述]
- **证据**：[代码片段 / 文档原文引用]
- **影响**：[会导致什么后果]
```

### 4. 让结论可争议

每个 agent 的结论应该是可被其他 agent 反驳的。如果所有 agent 都说"没问题"，大概率是审查失效而非方案完美。

## 操作模式

### 并行启动

子 agent 的任务独立，必须在同一条消息中同时发起所有 agent 调用。

### 领审员汇总：共识/分歧/盲区

拿到子 agent 的报告后，领审员按三层汇总：

| 层级 | 含义 | 处理方式 |
|------|------|---------|
| **共识** | 多个子 agent 指向同一个问题 | 高可信度，优先处理 |
| **分歧** | 子 agent 之间结论矛盾 | 需要深入判断或第二轮验证 |
| **盲区** | 没有任何子 agent 覆盖的维度 | 标注为"未审查"，不假设没问题 |

### 争议二轮验证（可选）

当子 agent 之间存在分歧时，领审员可以派一个新 agent 做定向验证：

```
"Agent 1 指出 [问题 X]。Agent 2 认为 [现有代码能处理]。
请验证：[写一个最小用例 / 读取具体代码确认 / 走查执行路径]。
结论基于验证结果，不基于推理。"
```

尽可能用代码执行验证，而非让 agent 再推理一遍。

## 维度设计框架

为新的审查场景设计维度时，按以下步骤思考：

1. **审查对象是什么？** 设计文档 / 实施计划 / 已完成代码 / 对话历史
2. **最危险的失败模式是什么？** 这决定了哪些维度最值得投入 agent
3. **每个维度需要什么输入？** 不同维度可能需要不同的文件和上下文
4. **维度之间是否有重叠？** 如果两个维度会看同一段代码/文档回答同一类问题，合并它们

示例——不同审查场景的维度切分：

| 审查场景 | 维度 1 | 维度 2 | 维度 3 |
|---------|--------|--------|--------|
| 设计文档审查 | 可行性质疑（能不能造出来） | 场景遗漏（没覆盖什么用户场景） | 技术债预判（会埋下什么坑） |
| 实施计划审查 | 算法正确性（逻辑对不对） | 代码集成（能不能融入现有系统） | 失败模式（生产环境怎么崩） |
| 已完成功能评估 | 需求覆盖缺口（缺了什么） | 回归风险（破坏了什么） | 性能瓶颈（慢在哪里） |

**这些只是示例。** 每个项目、每次审查应根据具体情况调整。

## 常见陷阱

1. **不要让 agent 写新方案。** agent 很爱"我建议重构成 XXX"，这会偏离审查目的。明确告诉它"只找问题，不提替代方案"
2. **不要只要"总体评价"。** 要求给出具体位置和具体证据
3. **不要让 agent 假设。** 它没有跑代码的能力，让它假设了就不可信。要求引用文档或代码原文
4. **控制 agent 数量。** 核心审查维度建议 2-4 个，超过 4 个汇总困难。如果维度本身是独立的（如设计审查的自洽性/完整性/合理性/RUBRIC 对齐），4 个全参与评分没问题。如果第 4 个是辅助性质（如 Slop 检测），可不参与评分

## 适用性判断

不是所有审查都适合对抗模式：

| 审查类型 | 是否适用 | 原因 |
|---------|---------|------|
| 设计审查 | ✅ 高度适用 | 未实现前是发现问题的最佳时机 |
| 方向评估 | ✅ 适用 | 子 agent 对抗找问题，领审员做决策 |
| 代码审查 | ✅ 适用 | 可以用具体代码行作为证据 |
| 安全扫描 | ⚠️ 部分适用 | 模式匹配部分不需要对抗，但安全风险评估可以 |
| 流程审计 | ❌ 对抗模式不适用 | 基于事实统计，不存在"立场"问题。但仍可用多智能体分工（按维度拆分，非对抗式） |
