# 测试标准 (Testing Standard)

> 本文件是测试相关术语和证据格式的**单一事实源(术语 SSoT)**。
> RUBRIC、finishing-rules、handoff 模板、evaluator 提示词若用到 Evidence Depth 词汇,**只引用不重复定义**。
> 术语修改:先改本文件,再同步下游(F3 文档先行)。

## 定位

本文件只做三件事:
1. 定义 **Evidence Depth(证据深度)** L1-L4 的语义
2. 规定 handoff 中 `## Evidence Depth` 字段的填写格式
3. 规定独立的 `## CI 阻断` 字段(证据强度之外的执行层能力)

**不在本文件范围**:
- "什么时候必须写测试"、"测试颗粒度"、"契约任务 vs 实现任务的测试要求" → 见 `docs/governance/testing-rules.md`
- 测试框架选型、覆盖率门槛、mock 策略 → 具体项目在项目 RUBRIC 或技术设计中定

---

## Evidence Depth 四级制

每一级定义的是**证据的强度**,不是测试的类型。可信度从低到高递增。

| L | 名称 | 含义 | 典型例子 |
|---|---|---|---|
| **L1** | 单元测试 | 隔离 mock,快速重复运行 | `npm test`、`pytest` 全绿 |
| **L2** | 冒烟测试 | 手动或自动走主路径(本地即真实,不强制部署) | 开发者本机跑一遍主路径;`npm run smoke`;staging 手工点一遍 |
| **L3** | 自动化 API 测试 | **脚本化**的接口/契约验证(不绑定"部署"概念,本地自动化 API 跑也算) | curl / postman / scripted HTTP 调用;API 集成测试 + 契约校验 |
| **L4** | 用户行为模拟 | **自动化工具**模拟真人端到端交互行为(任何形态) | Playwright 浏览器自动化;CLI pexpect 交互;移动端 Appium;键盘自动化 |

### 关键区分

- **L1 vs L2**:L1 是**隔离测试**(mock 依赖);L2 是**端到端冒烟**(真实模块协作,走主路径即可)
- **L2 vs L3**:L2 可以是手动(人跑一遍),L3 必须是**自动化 + 接口契约层**(脚本化,机器对机器)
- **L3 vs L4**:L3 测**接口契约**(请求/响应层),L4 测**用户行为**(模拟人的感知和操作,端到端体验)

---

## CI 阻断(独立字段,不是 Evidence Depth 的一层)

**"测试失败能否阻断流程"是执行层能力,不是证据强度**,因此从 Evidence Depth 剥离,改成独立字段。

`## CI 阻断` 字段填写:
- `✅` 已配置阻断机制:CI 失败阻断 merge / pre-commit hook exit 非 0 / PR check 强制通过
- `❌` 无阻断机制:失败只报告不阻断

### 为什么独立出来

同一层证据可能有阻断或没阻断,两个情况不应算作不同证据强度。比如 L3 通过 + CI 阻断 ≠ 更高层的 L3,而是"L3 证据 + 有阻断保证"。混在 Evidence Depth 里会让两个正交维度打架。

---

## 独立评级规则(重要)

**每一层独立评估,允许跳跃**。一个任务的证据可以只有 L1 和 L4,中间 L2/L3 为空。

**不允许**:
- 声称"到了 L3"而隐含"L1/L2 也通过"——这是递进式评级,不是本标准采用的独立评级
- 把"做了 L2"等同于"L1 也做了"——要分别声明

**理由**:
- 跳跃客观存在——比如一个纯浏览器项目可能只有 Playwright(L4)没有单元测试(L1)
- 递进式隐含会掩盖真实的证据缺口

## Handoff 字段格式

在 `docs/active/handoff.md` 中需要两个独立字段:`## Evidence Depth`(四层必须全列)和 `## CI 阻断`(独立标记)。

```markdown
## Evidence Depth

- L1 单元测试: ✅ `npm test` 全绿 (42 passed)
- L2 冒烟测试: ✅ 本地跑一遍主路径 OK
- L3 自动化 API 测试: ✅ `scripts/api-smoke.sh` 跑过 (12/12 绿)
- L4 用户行为模拟: ❌ 无

## CI 阻断

❌ 当前无 CI 阻断机制
```

### 格式规则

1. **四层必须全部列出**,没做的显式写 ❌
2. 每层状态有三种:
   - `✅` 已通过 + 简要证据(命令 / 文件路径 / 截图引用)
   - `❌ 未跑` 或 `❌ 无` 显式表示缺口
   - `⚠️ 部分` 只用于"某层只覆盖部分路径"的中间态,需附说明
3. 不允许只写层级不写证据——`✅` 后面必须有可核查的引用
4. `CI 阻断` 字段只有两种:`✅` + 简要说明机制 / `❌` + 一句理由
5. 两个字段任一为空等同于"未声明",hook 阻断 finishing(L3 实施时加 hook)

## 使用场景

### 谁在什么时候读本文件

- **implementation 阶段**:agent 准备动手写测试时,参考本文件决定"这次验证打算做到哪一层"
- **finishing 阶段**:agent 填 handoff 的 Evidence Depth 字段时,参考本文件格式
- **evaluator 运行时**:评估器的"测试充分性挑战者"参考本文件语义判断声明是否真实
- **用户审阅 handoff 时**:一眼看清"哪些层有、哪些层缺",不用猜测

### 本文件和 Superpowers TDD 的关系

- Superpowers TDD 负责**驱动写测试的过程**(写失败测试 → 实现 → 重构)
- 本文件负责**表达测试结果的强度**——TDD 产出的测试默认落在 L1 / L2,是否到 L3+ 要另行做
- 耦合边界:TDD skill 接口变动不影响 Evidence Depth 的分层概念(分层是抽象层,TDD 是执行层)
