# 系统设计文档模板

> 每个功能的系统设计保存到 `docs/superpowers/specs/[功能名]-design.md`。
> 本模板定义必须包含的结构。设计阶段的治理规则（docs/governance/design-rules.md）会逐节自检。
> 设计文档通过审查后才能进入 writing-plans。

---

## 使用方式

复制以下结构，逐节填写。每节填完后执行对应的自检问题。
以下代码示例使用 TypeScript 语法，**填写时请使用项目实际技术栈的语言**。

**规模适配**：design-rules.md 中有规模分级标准。轻量级需求只填第 1、8、9 节（精简版）；标准级不适用的节可写"不适用"；重量级所有节必填。**无论哪个级别，文档都必须先于代码存在。**

## 3. 接口定义

> 模块之间、模块与外部的所有交互点。

### 3.1 模块间接口

**[模块A] → [模块B]**

```typescript
// 函数签名（用项目的语言）

// 调用场景
// 在 [什么情况下] 由 [谁] 调用

// 错误处理
// 失败时返回/抛出什么，调用方怎么处理
```

[重复以上结构，每对交互一个]

### 3.2 外部接口（如有）

| 接口 | 方向 | 协议 | 数据格式 | 错误处理 |
|------|------|------|---------|---------|
| [API端点] | 入/出 | HTTP/WS/... | [格式] | [策略] |

### 3.3 前后端类型契约

> 如果项目有前后端分离，必须定义共享的类型契约。
> 如果项目是纯后端 API：本节改为定义"API 契约"（OpenAPI schema 或等效），确保 API 消费方和提供方有一致的类型约定。
> 如果本功能不涉及任何 API 端点变更：写"不适用——本功能不涉及 API 接口"。

**契约文件位置**：[按 ARCHITECTURE.md 的约定，如 `types/api/`]

> **技术栈适配**：
> - TypeScript monorepo：共享类型文件直接 import
> - OpenAPI / REST：以 OpenAPI schema 为契约源，前后端从 schema 生成类型
> - GraphQL：以 .graphql schema 为契约源
> - 跨语言（如 Python + React）：以 OpenAPI/JSON Schema 为契约源，各语言生成本地类型

```typescript
// 示例：用户注册接口的共享类型
// 前端和后端都从这个文件 import，不各自定义

interface RegisterRequest {
  email: string;
  password: string;
  displayName: string;   // 不是 display_name，不是 userName，统一用 displayName
}

interface RegisterResponse {
  userId: string;
  token: string;
}
```

**每个 API 端点**都需要在这里定义：
- 请求类型（Request）
- 响应类型（Response）
- 错误类型（如有特定的错误结构）

**自检**：
- [ ] 每个接口的双方都定义了？（不存在只有调用方没有实现方的接口）
- [ ] 参数类型和返回类型在项目类型系统中存在或已定义？
- [ ] 每个接口都有错误处理约定？（不是只考虑正常路径）
- [ ] 接口的入参和出参与需求摘要中的数据对得上？
- [ ] 接口是否简洁？（参数是否过多需要拆分？相似接口是否应合并？）
- [ ] **前后端共享的 API 类型是否都在契约文件中定义？**（不存在前端自己定义了一套类型而后端用另一套的情况）
- [ ] **字段命名是否统一？**（同一个概念在请求、响应、数据库中用同一个名字）

## 9. 全局自洽性检查

> 以上各节填完后，做最终的全局检查。

- [ ] **需求 ↔ 模块**：每个需求场景都能在模块划分中找到实现路径？
- [ ] **模块 ↔ 接口**：每个模块的职责都通过接口体现？没有模块是"孤岛"？
- [ ] **接口 ↔ 数据**：接口中使用的数据类型都在数据模型中定义了？
- [ ] **数据 ↔ 边界**：数据模型中每个字段的边界值都有处理？
- [ ] **依赖 ↔ 架构**：模块依赖方向符合 ARCHITECTURE.md？
- [ ] **决策 ↔ 需求**：设计决策没有偏离需求中的约束？
- [ ] **决策 ↔ 架构**：设计决策中的架构选择与 ARCHITECTURE.md 一致？
- [ ] **影响 ↔ 模块**：第 7 节改动的文件与第 2 节标记为"改动"的模块对应？
- [ ] **RUBRIC ↔ 设计**：第 6 节对每个 RUBRIC 惩罚项都有应对方式？
- [ ] **契约 ↔ 接口**：第 3.3 节的共享类型覆盖了所有 API 端点？字段命名在请求/响应/数据模型中一致？
