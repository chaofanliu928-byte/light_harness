# 决策: P0.9.1.5 写前流程阶段的触发条件

> 由 P0.9.1 spec(`docs/superpowers/specs/2026-04-17-p0-9-self-governance-design.md`)第七轮 fix-7 创建,**升 🟡 待用户决定**;第八轮用户拍板 → 🟢 已决定。

**状态**:🟢 已决定(2026-04-26 第八轮用户拍板)

**日期**:2026-04-26

**关联功能**:P0.9.1 self-governance(spec §1.3 边界 + §6.5)

## 问题

P0.9.1 spec §1.3 已锁定 P0.9.1.5(写前流程,吃自己狗粮)在 M0-M4 批次之前,但**具体触发时机未定**。即:P0.9.1 实施完成后到 P0.9.1.5 brainstorming 启动之间,什么情况下才进入 P0.9.1.5?是落地立即启动(强约束),还是用户在 M0-M4 启动前评估再决定(灵活)?

第七轮 spec /design-review 第二轮 D2 完整性挑战者点出此缺口。

## 方案

### 方案 A:P0.9.1 落地后立即启动(最严)

- **做法**:P0.9.1 commit 进 main 后,handoff / ROADMAP / SessionStart 立即引导启动 P0.9.1.5 brainstorming;不允许在 M0-M4 启动前跳过
- **优点**:
  - 闭环最严:吃自己狗粮的"先写新流程,再用新流程跑 M0-M4"顺序最清晰
  - 不留时序模糊空间(用户无需再评估"是否启动"),避免 P0.9.1.5 被无限延后
  - 与 §6.5 A6 兼容验证"P0.9.1 落地后用 P0.9.1.5 spec 走 P0.9.1 流程审查"自然衔接
- **缺点**:
  - 强制顺序可能与实际节奏冲突(用户可能想先看 M0-M4 中能否暴露 P0.9.1 自身缺陷,再决定 P0.9.1.5 设计方向)
  - 若 P0.9.1.5 难度被低估,会阻塞 M0-M4 启动(M0-M4 是 P0.9.1.5 落地后才用新流程跑的)

### 方案 B:M0-M4 启动前用户决定(灵活)

- **做法**:P0.9.1 落地后不强制启动 P0.9.1.5;用户在 M0-M4 启动前(brainstorming 阶段对接 M0)主动评估"P0.9.1.5 是否先做"。若决定先做,启 P0.9.1.5 brainstorming;若决定先跑 M0(用 P0.9.1 流程,不经 P0.9.1.5),M0 实战回流可能反过来定义 P0.9.1.5 需求
- **优点**:
  - 节奏灵活:用户可基于"P0.9.1 落地后 M0 启动前的实际状态"决定
  - 利用 M0 实战暴露 P0.9.1.5 真实需求(类似"先放出最小版,实战定义需求"),避免凭空设计
  - 不阻塞 M0-M4 启动
- **缺点**:
  - "吃自己狗粮"语义弱化:M0-M4 部分用 P0.9.1 流程跑(无 P0.9.1.5 写前流程),部分用 P0.9.1 + P0.9.1.5 双栈跑,顺序不一致
  - 用户每次评估增加认知负担(P0.9.1.5 阻塞 / 不阻塞 / 推迟)
  - 可能 P0.9.1.5 被无限推迟(如果 M0-M4 跑得顺,用户始终判"暂不需要")

### 方案 C:其他

- 候选 C1:P0.9.1 落地后**先反审本 spec**(见 fix-8 决策),反审产出 audit 后再判 P0.9.1.5 启动 — 但这把 fix-7 / fix-8 串成一条强链,增加耦合
- 候选 C2:P0.9.1 落地 + M0 brainstorming 阶段强制启动 P0.9.1.5(M0 触发,不是 P0.9.1 commit 触发)— 较 A 灵活但比 B 严

## designer 倾向(基于 spec 内逻辑,不替用户决定)

倾向 **方案 B(M0-M4 启动前用户决定,灵活)**,理由:

1. **§1.3 已锁定 P0.9.1.5 不在 P0.9.1 scope 内**,P0.9.1 spec 的责任是声明缺口存在 + 留 P0.9.1.5 占位,不是预定 P0.9.1.5 时序
2. **brainstorming 第二轮 Q3-B 拍板**:"用 P0.9.1 产出的 meta-review 流程做 P0.9.1.5"已隐含"P0.9.1 落地后视实际节奏推进",并未要求"立即启动"
3. **`feedback_judgment_basis` 原则**:无 P0.9.1 实战数据时,不应预先把 P0.9.1.5 触发条件锁死(类似 P0.9.2 不强行定可量化指标)
4. **避免与 fix-8 反审触发耦合**:反审本 spec(fix-8)和 P0.9.1.5 触发(fix-7)是两件事,B 让用户分别决策更清晰

但若用户偏好"严闭环优于灵活"(对应"feedback_design_philosophy 不过度简化、变更先改文档"原则),方案 A 可接受 — A 最不留模糊空间,符合"不便利化"原则。

## 决定

**选择**:**方案 B(M0-M4 启动前用户决定,灵活)**

**原因**:无 P0.9.1 实战数据时,不应预先把 P0.9.1.5 触发条件锁死,由 M0-M4 实战反向定义 P0.9.1.5 真实需求(`feedback_judgment_basis` 原则 — 禁止用凭空判断或别人项目数据支撑决策)。

## 后续影响

- **spec §1.3 边界声明**:P0.9.1.5 相关条目加 "B 触发:M0-M4 启动前用户评估" 表述
- **spec §6.5 A6 兼容验证**:P0.9.1.5 启动节奏表述同步 B 语义(brainstorming 在 M0 启动前由用户评估,实施阶段才触发 meta-review)
- **spec §8.3 P0.9.1.5 兼容验证**:对齐 B 触发节奏,启动前置由用户评估替代"P0.9.1 落地立即启动"
- **ROADMAP P0.9.1.5 行**:加触发条件 "M0-M4 启动前用户评估"
- **brainstorming-rules / handoff**:在 M0 brainstorming 启动前引导用户评估 P0.9.1.5 是否先做(实施层落地,P0.9.3 或 P0.9.1.5 自身实施时承接)
- **不影响 fix-8 反审触发**:fix-7(P0.9.1.5 触发)与 fix-8(反审本 spec)解耦,各自决策(参 fix-8 decision 文件)
